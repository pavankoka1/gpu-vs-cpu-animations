GPU vs CPU: five ways to animate the same UI

We rebuild the same betting strip in five different ways: from classic layout-driven animation to shader-based drawing. Each toggle does the same thing visually — open four spots into a 2×2 grid, or close them into a single row — but the browser does very different work under the hood.

Hands-on · about 12 minutes · best on mobile Safari or Chrome

In this article
  1. Width & height (layout / CPU)
  2. Transform & scale (compositor)
  3. Canvas 2D + CSS transform
  4. Canvas + offscreen supersample
  5. WebGL fragment shader
  6. Which approach when?

What you’re looking at. Four coloured “bet spots” sit on a fixed grid. Bets closed (1×4) shows one row of four narrow tiles. Bets open (2×2) expands them into a two-by-two layout. The animation uses the same duration, easing, and stagger everywhere so comparisons stay fair.

Why it matters. On low-end phones, animating width and height can force layout and paint every frame. Animating transform and opacity usually stays on the compositor and costs less. Canvas and WebGL add flexibility — gradients, games, effects — but you own pixels, timing, and how they interact with CSS.

Example 1 · Baseline

Animating width and height (layout on the CPU)

Here we change the real layout geometry of each tile: width, height, and transform for position. Every frame the engine may reflow siblings, recompute text, and repaint. That is the traditional “main thread” path: flexible and easy to reason about, but expensive when many things move at once.

What to watch. On a phone, hit the toggle during the animation and watch FPS dip. Stress mode runs all five demos; this one is usually the first to stutter.

Example 2 · Compositor

Transform and scale (GPU-friendly)

The layout box of each tile stays the same size as in the 1×4 strip. We only animate transform (move + scale) and border-radius. The browser promotes layers and interpolates matrices on the compositor; there is no per-frame layout for the size change.

Trade-off. You’re faking the 2×2 with scale and counter-scale on labels so text stays readable. That’s a common game UI trick: cheap motion, slightly more complex CSS.

Example 3 · Bitmap

Canvas 2D + the same CSS transform

Gradients are drawn with Canvas 2D into a bitmap; labels stay as DOM on top. The spots still move with transform, so motion stays compositor-friendly.

Catch. If the canvas only redraws once at the start or end of the transition while border-radius animates in CSS, corners and pixels fall out of sync — edges look torn. This demo redraws every animation frame with radii and timing matched to CSS so the bitmap tracks the rounded shape.

Example 4 · Supersampling

Canvas 2D with an offscreen 2× buffer

Same layout as example 3, but the rounded rect is drawn at double resolution into an offscreen canvas, then scaled down to the visible canvas with drawImage. That gives better coverage on diagonal edges when the layer is scaled non-uniformly in 2×2 mode.

Cost. More pixels to fill per frame than plain 2D, but still one composited layer and the same CSS motion path.

Example 5 · Shaders

WebGL: gradient + rounded rectangle in a fragment shader

The fill is computed per pixel: a signed distance field (SDF) for a rounded box, a diagonal linear gradient in canvas space, and smooth anti-aliasing at the edge. Radii and buffer size follow the same CSS scale as the DOM so corners stay aligned while the strip animates between 1×4 and 2×2.

Why bother. You get procedural control (effects, video textures, particles) and predictable edges at the cost of shader code, context setup, and fallbacks for older devices.

How to read the meters. FPS and frame time update every second. Toggle while an animation is running; cheap approaches hold steady, expensive ones hitch. On iOS Safari the gap between example 1 and 2–5 is usually the clearest. Chrome’s device toolbar is a useful second choice.

Under the hood (short). The browser pipeline splits roughly into style → layout → paint → composite. width/height changes often pull you back into layout. transform and opacity on a promoted layer mostly skip to compositing. Canvas and WebGL hand you a bitmap the compositor treats as a texture — your job is to keep that bitmap consistent with anything CSS animates beside it.

Which approach should you pick?

There is no single winner — it depends on whether you care most about simplicity, lowest jank, pixel effects, or team skills. Use this as a rule of thumb, then measure on your target devices.

Layout must be real Width / height animation — when something actually needs to reflow (copy wrapping, flex/grid resolving, screen readers seeing true geometry) and the moving region is small or rare. Avoid as the main motion tool for large, always-on UI on weak devices.
Default UI motion Transform + opacity — panels, drawers, HUD blocks, anything that can live on a compositor layer. Best trade-off of smoothness and authoring cost. Use will-change sparingly; don’t mix heavy layout animation on the same subtree.
Dynamic 2D pixels Canvas 2D — charts, sprites, scratch areas, gradients you draw every frame. Keep DPR, resize, and raster/CSS timing in mind when transforms animate beside the bitmap.
Softer 2D edges Offscreen supersample — same 2D API; draw big, blit small when CSS scales the canvas and edges look rough. Costs memory and fill rate.
Shaders & effects WebGL — lighting, blur passes, particles, custom AA. Heavier setup; test context loss and mobile GPUs.

Bottom line. Start with CSS transform and opacity for motion. Reach for layout animation only when the layout itself must change. Add Canvas when pixels are dynamic; add WebGL when the GPU program is the product. Measure on real hardware — this page is a lab, not a substitute for profiling your app.
